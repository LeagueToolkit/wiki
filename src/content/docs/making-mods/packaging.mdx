LeagueToolkit ships two mod package formats: `.modpkg` and `.fantome`. Both are fully supported. The right choice comes down to whether you want every project feature or the widest reach.

:::tip[Publishing to a mod site? Use fantome.]
`.modpkg` is the more capable format, but **no mod distribution website fully supports it yet**. If you upload one today it will not reach most of your audience.

Until that changes, package as `.fantome` when you publish through a centralized platform such as [Runeforge](https://runeforge.dev/). Reach for `.modpkg` when you distribute outside one - your own GitHub releases, a direct download, or a Discord server - where nothing sits between you and the person installing it.

This is a limitation of platform support, not of the format. Nothing about your project changes: the [Creator Workshop](/making-mods/workshop/) and [`league-mod`](/making-mods/command-line/) both pack either format from the same project.
:::

## modpkg

The `.modpkg` format is the modern container, and the one we recommend authoring against. It's purpose-built for the LeagueToolkit launcher and exposes the full surface area of the project model. Note that authoring and publishing are separate decisions - see [Choosing a format](#choosing-a-format) for what to actually upload.

### Structure

A modpkg file is a binary archive containing:

- **Metadata**: name, version, authors, description, tags, champions, maps
- **Thumbnail**: optional preview image (webp or png)
- **Layers**: one or more content layers, each containing game files
- **Configuration**: layer priorities, string overrides, and other settings

### Why it's the default

- Full project metadata, including authors with roles and rich tagging
- Multi-layer composition with configurable priorities
- Per-layer string overrides
- Embedded thumbnails
- Binary layout aligned with the WAD chunk format, so the launcher can copy chunks straight into the overlay WAD without re-encoding

## fantome

The `.fantome` format comes from [Fantome](https://github.com/LeagueToolkit/Fantome), the mod manager the LeagueToolkit organization built under its original name; cslol-manager carried the format forward, and it is still what most League mods are packaged as. ltk-manager supports it fully: import, export, and library management all work the same as for modpkg, and it reads metadata fields beyond the original cslol-manager specification, so existing libraries keep working as they are.

### When to choose fantome

Pick fantome when reach matters more than the latest project features - which today means **any time you publish through a mod distribution site**. Mods packaged as `.fantome` work in cslol-manager and across the existing ecosystem of tools and mod sites, which is invaluable while users migrate to ltk-manager and while those sites add `.modpkg` support.

### Importing a fantome into the Workshop

Installing a `.fantome` and importing one are different operations, and it is
worth keeping them apart:

- **Installing** adds the archive to your library to play with. The file stays a
  fantome - see [Managing Your Library](/mod-management/library/).
- **Importing** turns the archive into an editable Workshop project, so you can
  change it and repack it.

On import, the Workshop:

1. Reads the archive's `META/info.json` for name, version, author, and description
2. Extracts each WAD in the archive into the new project's content layer
3. Pulls in the archive's readme, if it has one
4. Writes the result out as a standard Workshop project

You can preview an archive's metadata and WAD list before committing to the
import. See [Creating a Project](/making-mods/project/#from-fantome).

The bridging is best-effort: not every modpkg feature has a fantome equivalent, so imported projects may need a light pass in the editor before they round-trip cleanly.

### Trade-offs vs modpkg

The fantome format is a renamed ZIP archive - the [Fantome package reference](/reference/mod-packages/fantome/) documents the layout in full. That's its strength for portability and its weakness for runtime cost. Compared to modpkg, fantome:

- Is single-layer by design
- Has no concept of per-layer string overrides
- Requires the launcher to decompress and repack ZIP entries when building the overlay, instead of the direct chunk copy that modpkg enables

These are limitations of the format, not the implementation: ltk-manager supports everything fantome itself can express.

## Choosing a format

Both formats are packed from the same project by the
[Creator Workshop](/making-mods/workshop/) and by
[`league-mod pack`](/making-mods/command-line/#pack), so you never have to
commit upfront. Decide per release, by where the mod is going:

| Where you're publishing                                             | Package as               |
| ------------------------------------------------------------------- | ------------------------ |
| A mod distribution site such as [Runeforge](https://runeforge.dev/) | `.fantome`               |
| Your own GitHub releases, a direct download, or a Discord server    | `.modpkg`                |
| Users who are still on cslol-manager                                | `.fantome`               |
| Not sure                                                            | `.fantome`, or ship both |

The reason is reach, not quality. `.modpkg` is the better format on every
technical axis above, but no distribution site fully supports it yet, so a
`.modpkg` upload reaches far fewer people than the same mod packaged as
`.fantome`.

Shipping both is a reasonable default while that gap closes: author in
`.modpkg`, export a `.fantome` alongside it, and let people take whichever
their tools handle. We will update this page when platform support lands.
