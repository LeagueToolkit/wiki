LeagueToolkit ships two mod package formats: `.modpkg` and `.fantome`. Both are first-class citizens in the ecosystem. The right choice comes down to whether you want the full project surface or maximum reach across the wider modding community.

## modpkg

The `.modpkg` format is the modern container we recommend for new projects. It's purpose-built for the LeagueToolkit launcher and exposes the full surface area of the project model.

### Structure

A modpkg file is a binary archive containing:

- **Metadata**: name, version, authors, description, tags, champions, maps
- **Thumbnail**: optional preview image (webp or png)
- **Layers**: one or more content layers, each containing game files
- **Configuration**: layer priorities, string overrides, and other settings

### Why it's the default

- Full project metadata, including authors with roles and rich tagging
- Multi-layer composition with configurable priorities
- Per-layer string overrides
- Embedded thumbnails
- Binary layout aligned with the WAD chunk format, so the launcher can copy chunks straight into the overlay WAD without re-encoding

## fantome

The `.fantome` format originated in cslol-manager and remains the lingua franca of the League modding community. ltk-manager provides first-class support for it: import, export, and library management all work the same as for modpkg, and the implementation handles metadata fields beyond the original cslol-manager specification, so existing libraries continue to work without conversion.

### When to choose fantome

Pick fantome when reach matters more than the latest project features. Mods packaged as `.fantome` work in cslol-manager and across the existing ecosystem of tools and mod sites, which is invaluable while users migrate to ltk-manager.

### Behavior on import

When you import a `.fantome` file, ltk-manager treats it as a real project and bridges it onto the modern model:

1. Extracts the ZIP archive contents
2. Parses the WAD file structure
3. Converts the content into the internal mod representation, lifting in modpkg capabilities where they map cleanly
4. Stores it as a standard mod in the library

The bridging is best-effort: not every modpkg feature has a fantome equivalent, so converted mods may need a light pass in the editor before they round-trip cleanly.

### Trade-offs vs modpkg

The fantome format is a renamed ZIP archive. That's its strength for portability and its weakness for runtime cost. Compared to modpkg, fantome:

- Is single-layer by design
- Has no concept of per-layer string overrides
- Requires the launcher to decompress and repack ZIP entries when building the overlay, instead of the direct chunk copy that modpkg enables

These are limitations of the format, not the implementation: ltk-manager supports everything fantome itself can express.

## Choosing a format

Both formats can be produced from the [Creator Workshop](/guides/mod-creation/workshop-overview/) pack feature, so you don't have to commit upfront. A common pattern is to author in modpkg and export a fantome alongside it for distribution to users on older tools.
